<task type="auto">
  <name>Task 2: Fix GET_STATE timeout -- eliminate race between STATE_UPDATED push and sendViaPort listener</name>
  <files>
    apps/extension/entrypoints/background.ts
    apps/extension/src/messaging/send.ts
  </files>
  <action>
ROOT CAUSE: When sidepanel connects, background immediately pushes STATE_UPDATED (lines 274-277) via sendStateToSidepanel. Meanwhile, sidepanel calls sendViaPort for GET_STATE which adds a listener filtering for `type === 'RESPONSE' && requestType === 'GET_STATE'`. The STATE_UPDATED message arrives at the sendViaPort listener first but does NOT match (it's type 'STATE_UPDATED', not 'RESPONSE'), so it's correctly ignored. However, the real issue is that sendStateToSidepanel calls checkMappingForTab internally (via the sendStateToSidepanel -> buildState path), which may trigger an async chain that delays the GET_STATE response, or the initial STATE_UPDATED push may cause background to process messages out of order.

The robust fix has two parts:

**Part A -- background.ts: Remove immediate STATE_UPDATED push on connect (lines 274-277)**

Replace:
```typescript
// Push current tab state to newly connected sidepanel
if (activeTabId !== null) {
  sendStateToSidepanel(activeTabId);
}
```

With:
```typescript
// Do NOT push state immediately on connect.
// Sidepanel will request state via GET_STATE, which is the canonical
// way to get initial state. The immediate push caused race conditions
// where async work from sendStateToSidepanel interfered with the
// GET_STATE handler running concurrently.
```

This is safe because:
- Sidepanel always sends GET_STATE immediately after connecting (App.tsx line 66)
- GET_STATE handler already calls checkMappingForTab and buildState
- Removing the redundant push eliminates the race entirely

**Part B -- background.ts: Ensure activeTabId is initialized before GET_STATE runs**

In the GET_STATE handler (line 286), `activeTabId` may be null on fresh extension load if `tabs.onActivated` hasn't fired yet. Add a query to initialize it:

Replace lines 290-299:
```typescript
case 'GET_STATE': {
  console.log('[Background] GET_STATE: Refreshing mapping detection...');
  if (activeTabId !== null) {
    console.log('[Background] GET_STATE: Checking mapping for tab:', activeTabId);
    await checkMappingForTab(activeTabId);
  }
  const state = await buildState(currentTabId);
  ...
```

With:
```typescript
case 'GET_STATE': {
  // Ensure activeTabId is known (may be null on fresh extension load)
  if (activeTabId === null) {
    const [activeTab] = await browser.tabs.query({ active: true, currentWindow: true });
    if (activeTab?.id) {
      activeTabId = activeTab.id;
    }
  }
  const resolvedTabId = activeTabId ?? 0;
  console.log('[Background] GET_STATE: Refreshing mapping detection for tab:', resolvedTabId);
  if (resolvedTabId > 0) {
    await checkMappingForTab(resolvedTabId);
  }
  const state = await buildState(resolvedTabId);
  console.log('[Background] GET_STATE: Returning state, hasMapping:', state.hasMapping, 'mappingId:', state.mappingId, 'availableMappings:', state.availableMappings.length);
  port.postMessage({ type: 'RESPONSE', requestType: 'GET_STATE', success: true, data: state });
  break;
}
```

Also update `const currentTabId = activeTabId ?? 0;` (line 286) to be set AFTER the GET_STATE case handles tab initialization. Move it to after the switch or compute it per-case. The simplest approach: keep `currentTabId` at line 286 but use `resolvedTabId` only in the GET_STATE case (other cases already have their own activeTabId checks).

**Part C -- send.ts: No changes needed** (sendViaPort correctly adds listener before posting, and filters by type/requestType -- confirmed by debug session evidence at timestamp 12:03)
