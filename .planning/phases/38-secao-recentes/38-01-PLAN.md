```typescript
export interface RecentRowEntry {
  /** 0-based row index within the batch */
  rowIndex: number;
  /** Display identifier value (primary from batch settings, or first column fallback) */
  identifierValue: string | null;
  /** Column name of the identifier (for tooltip display) */
  identifierFieldKey: string | null;
  /** Fill result status: 'success' | 'failed' | 'navigated' (default) */
  status: 'success' | 'failed' | 'navigated';
  /** Timestamp when row was last visited (for ordering) */
  visitedAt: number;
}

export interface RecentRowsState {
  /** Map of batchId to array of recent row entries (max 10 per batch) */
  byBatch: Record<string, RecentRowEntry[]>;
}

export const DEFAULT_RECENT_ROWS: RecentRowsState = {
  byBatch: {},
};
```

2. Create `apps/extension/src/storage/recentes.ts` following the same pattern as `preferences.ts` and `selection.ts`:
   - Use `storage.defineItem<RecentRowsState>('local:populatte:recentes', { fallback: DEFAULT_RECENT_ROWS })`
   - Implement these methods on `recentRowsStorage`:
     - `getEntries(batchId: string): Promise<RecentRowEntry[]>` - returns entries for batch, most recent first
     - `addEntry(batchId: string, entry: Omit<RecentRowEntry, 'visitedAt'>): Promise<void>` - adds entry with deduplication (if rowIndex already exists, remove old one first, then prepend new one). Enforce max 10 entries per batch with FIFO eviction (remove oldest when at capacity). Set visitedAt to Date.now().
     - `updateStatus(batchId: string, rowIndex: number, status: 'success' | 'failed'): Promise<void>` - updates the status field of an existing entry matching rowIndex. No-op if entry not found.
     - `clearForBatch(batchId: string): Promise<void>` - removes all entries for a batch
     - `getState(): Promise<RecentRowsState>` - raw state access
   - All methods follow the try/catch + console.error pattern established in other storage modules.

3. In `apps/extension/entrypoints/sidepanel/App.tsx`:
   - Import `recentRowsStorage` from `../../src/storage/recentes`
   - Add state: `const [recentRows, setRecentRows] = useState<RecentRowEntry[]>([])` (import RecentRowEntry from storage/types)
   - Create a helper function `addToRecent(rowIndex: number)` that:
     a. Gets identifier value from `state.identifierPrimary` (fallback: null — the fallback to first column is handled later in the UI component when rendering)
     b. Calls `await recentRowsStorage.addEntry(state.batchId!, { rowIndex, identifierValue: state.identifierPrimary ?? null, identifierFieldKey: state.identifierFieldKey ?? null, status: 'navigated' })`
     c. Refreshes local state: `const entries = await recentRowsStorage.getEntries(state.batchId!); setRecentRows(entries);`
   - Wire `addToRecent` calls:
     a. In `handleNext()`: after the `await sendViaPort(...)` call, add `await addToRecent(state.rowIndex + 1)` (use the new row index, which will be state.rowIndex + 1 clamped to max). Actually, more reliable: after the state comes back via STATE_UPDATED, the rowIndex will be updated. Instead, add the call right after sendViaPort resolves since we know the navigation succeeded. Use `Math.min(state.rowIndex + 1, state.rowTotal - 1)` as the rowIndex to add.
     b. In `handlePrev()`: same pattern, `Math.max(state.rowIndex - 1, 0)`
     c. In `handleFill()`: after the fill completes and we have results, call `await recentRowsStorage.updateStatus(state.batchId!, state.rowIndex, allSuccess ? 'success' : 'failed')` where `allSuccess` is determined from the step results. Then refresh: `const entries = await recentRowsStorage.getEntries(state.batchId!); setRecentRows(entries);`
   - Wire initial load: Add a `useEffect` that watches `state?.batchId`. When batchId changes and is not null, load recent rows: `recentRowsStorage.getEntries(batchId).then(setRecentRows)`. When batchId is null, set `setRecentRows([])`.
   - Also add the CURRENT row to recent when batch first loads (the initial row 0). In the same useEffect: after loading entries, check if current rowIndex (from state) is already in the list. If not, add it via `addToRecent(state.rowIndex)`.
   - NOTE: The `addToRecent` helper needs `state.batchId` to be non-null. Guard accordingly.

Key implementation note: The `identifierPrimary` value in state may not yet be available when we call addToRecent (if state hasn't updated). Use the value from `state` at the time of the call — it represents the row we were ON, and after navigation the state updates. For handleNext/handlePrev, we're recording the NEW row we navigate TO. However, the identifier for the NEW row isn't available yet (state updates asynchronously). Solution: record with `identifierValue: null` initially, then update it when the STATE_UPDATED message arrives with the new identifier. Add logic in the STATE_UPDATED listener: when rowIndex changes, update the recent entry's identifier if it was null. Actually, simpler approach: in the STATE_UPDATED handler inside the messageListener, after `setState(newState)`, if `newState.batchId` and `newState.identifierPrimary`, call `recentRowsStorage.addEntry(newState.batchId, { rowIndex: newState.rowIndex, identifierValue: newState.identifierPrimary, identifierFieldKey: newState.identifierFieldKey, status: 'navigated' })` and refresh recentRows. This way, every state update that changes the row automatically records the new row with correct identifier. Remove the explicit `addToRecent` calls from handleNext/handlePrev to avoid double-recording.

REVISED APPROACH (cleaner):
- Track previous rowIndex with a ref: `const prevRowIndexRef = useRef<number | null>(null)`
- In the STATE_UPDATED handler (messageListener function), after `setState(newState)`:
  - If `newState.batchId && newState.rowIndex !== prevRowIndexRef.current`:
    - Call `recentRowsStorage.addEntry(newState.batchId, { rowIndex: newState.rowIndex, identifierValue: newState.identifierPrimary ?? null, identifierFieldKey: newState.identifierFieldKey ?? null, status: 'navigated' }).then(() => recentRowsStorage.getEntries(newState.batchId!).then(setRecentRows))`
    - Update `prevRowIndexRef.current = newState.rowIndex`
  - This handles ALL navigation sources (next, prev, ROW_SELECT from Recentes click) uniformly.
- In handleFill: after fill completes, update status via `recentRowsStorage.updateStatus(...)` and refresh recentRows state.
- Initial load: in the batchId useEffect, load entries and set prevRowIndexRef.current to state.rowIndex.

This approach ensures:
- Every row navigation is tracked via STATE_UPDATED (single source of truth)
- Identifiers are always available (state has been resolved by background)
- Fill status updates happen after fill completes
- No duplicate recording
